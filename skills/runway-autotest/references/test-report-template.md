生成测试报告时，严格按此模版输出 Markdown 内容。

---

## 模版结构

```markdown
# {模块名} 接口测试报告

> 执行时间：{yyyy-MM-dd HH:mm} · 环境：{Base URL} · 执行人：Claude Code 自动生成
```

## 填写规则

### 总览表

- `total` = 单接口用例数 + E2E 步骤总数
- `pass` / `fail` / `skip` 统计单接口用例 + E2E 步骤，全部汇总
- 通过率 = pass / total，保留整数百分比
- 全部通过时显示 🎉 那行；存在失败时显示 ⚠️ 那行，二选一

### 用例明细表

- **实际请求体**列：填替换占位符后的**实际值**，方便直接复现；无入参填 `无`；Skip 用例填 `—`
- **实际响应摘要**列：只写关键字段（code 值 + 关键业务字段），不贴完整 JSON
- **TraceId**：从响应 header `M-TraceId` 提取；所有实际执行的用例（Pass/Fail）必须填写，服务端未返回则填 `（未返回）`；仅 Skip（未执行）用例填 `—`
- 通过用例和失败/跳过用例分两个子表
- **失败/跳过子表范围**：同时包含单接口用例和 E2E 联合用例中的失败/跳过步骤，统一汇总在此
- **根因 & 建议**列：失败用例必须给出根因分析和修复建议，不能只描述现象；Skip 用例填 `—`

### E2E 联合用例章节

- 若无 E2E 用例，整个章节省略
- 每个 E2E 场景单独一个子表，标题为 `### E2E-{序号} {场景名称}`
- 步骤全部通过时，整体结论为 Pass；任一步骤 Fail，后续步骤标记 Skip，整体为 Fail
- E2E 中的失败/跳过步骤**同时**在「失败/跳过用例」子表中汇总，编号格式为 `E2E-{序号}-Step{N}`

### 结论与建议章节

- 明确区分「需修复」和「可接受/暂不处理」两类
- 每条需修复项必须对应一个具体的 TC 编号
- 若全部通过，直接写一句话结论，不需要分类列表
