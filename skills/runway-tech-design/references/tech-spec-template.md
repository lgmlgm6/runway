## 一、背景与目标

> 本层回答：**为什么改、要达成什么目标、边界在哪里。**
> reviewer 读完本层应能快速理解问题背景、设计目标与整体范围。

### 需求背景

{1-2 句话：当前问题、需求来源、要解决的核心矛盾}

### 设计目标

- {目标 1}
- {目标 2}

### 范围边界

- **本次包含：** {范围}
- **本次不包含：** {明确排除项}

### 关键信息来源

- {需求规格 / 现有代码 / 用户确认 / 其他约束}

### 总体视图（按需）

> **simple 方案**：可省略。
> **complex 方案**：建议至少提供一张 Mermaid 图，帮助 reviewer 建立整体心智模型。

```mermaid
{架构图 / 流程图 / 时序图}
```

---

{这个模块为什么改，在整体方案中承担什么职责}

**设计说明：**
{核心逻辑，状态流转，写入路径，失败处理，幂等保证}

> - 存在复杂状态流转
> - 存在异步链路（消息 / 重试 / 补偿）
> - 涉及多存储协同
> - 仅凭文字 reviewer 难以理解主链路

```mermaid

```

## 自检清单

提交给用户前：

- [ ] 一到六必填章节均已填写；`七、架构ADR` 仅在 ADR 触发时保留
- [ ] 未出现 `方案摘要`、`设计原则`、`发布与风险控制` 独立章节
- [ ] `二、详细设计` 只写实现方案、业务逻辑、关键流程、状态变化、模块边界
- [ ] `三、接口协议变更` 只写对外请求/响应或契约变化、兼容性说明，与详细设计无重复叙述，且未写入内部 RPC / 内部事件 / 模块内参数细节
- [ ] complex 方案：总体视图已提供足够全局信息，必要时已补 Mermaid 图
- [ ] 每个模块：满足补图条件时，已在模块内补图
- [ ] 基础设施各章节：涉及则填写，不涉及明确写原因，无留空
- [ ] 验证策略包含`覆盖风险`列，并覆盖关键实现风险或兼容性风险
- [ ] 若存在 `七、架构ADR`，表格已写清`方案对比`、`选型依据`、`决策理由`
- [ ] 无 TBD、待定、后续确认等模糊占位符
