# 测试用例文档模版

生成测试用例文档时，严格按此模版输出 Markdown 内容。

---

## 模版结构

```markdown
# {模块名} - 测试用例

> 基于 API 设计文档 + PRD 生成，覆盖正常流程、入参校验、业务规则三类场景，并包含端到端联合用例。
>
> **测试环境**：{Base URL}
>
> **用例总数**：单接口 {tc_total} 条 + E2E {e2e_total} 步，共 {total} 条

---

## 接口{序号}：{接口名称}

`{METHOD} {PATH}`

| 编号 | 场景类型 | 场景描述 | 请求体 | 预期结果 |
|------------|----------|----------------------|-----------------------------------|----------------------|
| TC-{i}-1 | 正常流程 | {场景描述} | pageNum=1, pageSize=10, userId={test_user_id} | {预期结果断言} |
| TC-{i}-{n} | 入参校验 | {场景描述} | pageNum=1, pageSize=10 | code≠0，msg 非空 |
| TC-{i}-{n} | 业务规则 | {场景描述} | pageNum=1, userId=非法格式 | {预期结果断言} |

---

## E2E 端到端用例

### E2E-{序号} {场景名称}

> {一句话描述验证目标}

| 编号 | 接口 | 请求体 | 预期结果 |
|----------------|-------------------------------|-----------------------------------|----------------------|
| E2E-{n}-Step1 | {METHOD} {path} | {字段}={值}, {字段}={占位符} | {预期结果} |
| E2E-{n}-Step2 | {METHOD} {path} | {字段}={值} | {预期结果}，存入 {Step1.字段名} |
| E2E-{n}-Step3 | {METHOD} {path} | {字段}={值} | {预期结果}，{字段}={Step2.字段名} |
```

---

## 填写规则

### 用例总数

- 写在文档头部 `>` 引用块中，格式：`单接口 {tc_total} 条 + E2E {e2e_total} 步，共 {total} 条`
- 生成完所有用例后统计填入，E2E 按步骤数计算

### 请求体写法

- **禁止使用反引号包裹请求体**，不要写成 `` `{"key":"value"}` ``，直接写 `key=value, key2=value2` 格式
- **所有不确定的动态值必须用占位符**，不写死具体 ID 或业务数据，格式：`key={占位符名}`
- 已知的固定参数直接写具体值，如 `pageNum=1, pageSize=10`
- 无入参的接口写 `无`
- 多个字段用逗号分隔，如 `pageNum=1, pageSize=10, userId={test_user_id}`
- 超长列表写 `ids=[{id1},{id2},...]` 省略中间部分

### 预期结果写法

- 用键值断言格式，如 `code=0`、`code≠0`、`data.list.length=1`
- 多个断言用 `，` 分隔，如 `code=0，data.operator 非空，data.createTime 格式为 yyyy-MM-dd HH:mm`
- 涉及步骤间变量引用时写 `{字段}={StepN.字段名}`

### 场景类型列

- 每个接口用一张表，「场景类型」作为独立列，取值：`正常流程` / `入参校验` / `业务规则`
- 行内顺序按 **正常流程 → 入参校验 → 业务规则** 排列
- 若某类场景无用例，直接不写该类型的行，无需任何占位
