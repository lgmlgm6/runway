## 验收标准（Acceptance Criteria）

> AC 编号格式：`AC-01`、`AC-02` …，全文唯一，供 Stage 3 生成 TC 时引用。
> 优先级：P0 = 核心路径，必须覆盖；P1 = 重要但非阻断。
> Then 必须是可验证的具体结果，不允许模糊描述（如"正常展示"、"不报错"）。

| AC编号 | Given（前置条件） | When（用户操作/系统触发） | Then（可验证的预期结果） | 覆盖需求 | 优先级 |
|--------|-----------------|------------------------|------------------------|---------|--------|
| AC-01  | {前置条件} | {用户操作} | {可验证的具体结果} | F1 | P0 |
| AC-01  | 订单状态为"已发货"，配送员已接单 | 用户打开订单详情页 | 页面展示配送员实时位置地图，位置误差 < 100m | F1 | P0 |
| AC-02  | 订单状态为"待发货" | 用户打开订单详情页 | 地图区域不展示，显示"暂无配送信息" | F1 | P0 |

## 自检清单

提交给用户前：

- [ ] `已确认`、`假设`、`待确认` 三类均已填写且分类正确
- [ ] 每条假设都是必要的，且明确标注为假设
- [ ] 每条待确认项都有影响说明、负责人、需要时间
- [ ] 无 `TBD`、`待定`、`后续确认` 等模糊占位符
- [ ] 每条功能需求至少对应一条 AC，AC 编号唯一（AC-01、AC-02…）
- [ ] 每条 AC 的 Then 是可验证的具体结果，不是"正常展示"、"不报错"等模糊描述
- [ ] P0 AC 覆盖了核心路径和主要异常路径
- [ ] "不做"列表明确列出了容易被误解为要做的内容
- [ ] 每条验收标准只有一种解读
- [ ] 歧义分数和维度得分已记录
- [ ] 若歧义仍高于阈值，规格中已明确标注哪些项需在推进前确认
